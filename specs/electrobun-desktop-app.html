Purpose

Package pi-browser-dashboard — today a Vite/React SPA (apps/web) talking to a Bun/Hono daemon (apps/daemon, port 8787) — as a single cross-platform Electrobun desktop application, and bundle it in CI so users can download installers from GitHub Releases. Electrobun runs the main process on Bun and renders through the OS-native webview, yielding ~14 MB bundles with built-in incremental updates.

Problem

The dashboard currently requires a developer to run two processes (bun run dev launches daemon + Vite) and open a browser. There is no installable artifact. The browser deployment also leans on a Cloudflare quick-tunnel and a same-origin /__api Vite proxy (see apps/web/src/lib/apiBase.ts) — neither exists in a packaged app, so the SPA-to-daemon transport and CORS must be re-wired for a native webview origin. The daemon's main.ts is a top-level-await script that auto-starts a tunnel; it cannot be cleanly embedded as-is.

Solution

Add an apps/desktop Electrobun package. Its Bun main process boots the daemon in-process (via a new startDaemon() export) with the tunnel disabled and CORS opened for the webview origin, then opens a BrowserWindow loading the production SPA build via the views:// protocol. The SPA is built with VITE_API_URL=http://localhost:8787 so apiBase()/wsBase() hit the embedded daemon directly (bypassing the dev-only /__api proxy). A new desktop-release.yml GitHub Actions workflow runs electrobun build on a per-OS matrix (macOS / Windows / Linux) and attaches the artifacts to a GitHub Release on tag push.

Questionables

Embed the daemon in-process vs. spawn it as a subprocess?

Decision: in-process via startDaemon(). Single Bun runtime, single bundle, easiest lifecycle (no orphaned child on crash). The extraction is independently useful and testable. Fallback to Bun.spawn only if Electrobun's main-process sandbox blocks Bun.serve.

Will terminal/canvas WebSockets work from a views:// origin to ws://localhost:8787?

Likely yes — native webviews permit ws://localhost and WS handshakes are not blocked by CORS the same way. Mixed-content (secure scheme → insecure ws) is the risk. Validated manually in Phase 4; mitigation if blocked: serve the daemon over ws://127.0.0.1 with an allow-list, or have the daemon serve the SPA itself (same-origin) instead of views://.

Exact webview Origin header value?

Electrobun's custom scheme emits an origin like views://mainview (confirm at runtime via daemon request log). Plan handles this by injecting the origin into CORS from desktopEnv.ts, plus a views://-scheme allowance in the daemon CORS callback so a minor string mismatch doesn't break it.

Code-signing / notarization?

Out of scope for v1. Ship unsigned artifacts (users right-click-open on macOS, SmartScreen prompt on Windows). Signing keys (APPLE_*, Windows cert) added later as repo secrets + config — noted as future work, not a blocker for "download from GitHub".

Auto-update baseUrl — GitHub Releases?

Set release.baseUrl to the repo's GitHub Releases download URL so Electrobun's updater can pull update.json + patches. Updater is configured but not a v1 acceptance criterion; the acceptance bar is a downloadable install.
