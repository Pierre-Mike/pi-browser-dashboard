PRs: pid-apps-authoring Phase 1–5 (not yet created)

Purpose

pid-apps (specs/pid-html-extensions.html) shipped a way to view HTML dropped into a project's .pid/ folder. This plan closes the three gaps left between "I have an idea" and "I see it running," and between "I see it running" and "it has a backend": an in-dashboard way to create a pid-app instead of hand-authoring files on disk, zero-config visibility for the specs/*.html plans a project already has (this very file included — see Dogfooding), and a one-command path to graduate a pid-app into a full manifest-based extension with RPC capabilities. All three are additive over the existing security and rendering primitives — no new attack-surface class, and the first two need no daemon restart.

Problem

Two capable systems already ship in this repo and neither talks to the other. pid-apps (PRs #320-#327) discovers .pid/<id>/index.html on demand and renders it in a sandboxed tab — but it is read-only: an app is hand-created on disk, and a project's own specs/*.html plans are invisible until someone manually copies one into .pid/. The extension platform (PR #116 onward) is the opposite shape: manifest-based, capability-gated, has a CLI scaffold (pid-ext) — but it is boot-time (a new extension needs a daemon restart) and entirely disconnected from pid-apps, so a user who outgrows a static pid-app has no path forward except hand-writing a manifest.json from scratch.

Solution

Add three thin, additive lanes over the existing pipeline, none of which touch the security primitives (realpath containment, CSP, sandboxed iframe) already proven in pid-apps: (1) create — a POST /projects/:id/pid-apps route plus a dashboard "+ New App" control that scaffolds .pid/<name>/index.html the same way pid-settings already writes project state (tmp+rename); (2) discover — a second, read-only zero-config root, <project>/specs/*.html, merged into the same listApps/resolveAsset pipeline, so every plan a project already has shows up as a pidapp: tab with no copy step; (3) promote — a pid-promote CLI, mirroring pid-ext's shape exactly, that moves a .pid/<id>/ pid-app into .pid/extensions/<id>/ with a generated manifest.json, reusing the exact manifest builder pid-ext already has.

Questionables open decisions

Surfaced rather than silently decided. Each has a recommended default; confirm before the affected phase.

1. Unify specs/*.html into pid-apps (a source discriminant) or ship a separate spec-apps feature?

Recommend: unify. The serving pipeline — realpath containment, CSP, mime, sandboxed iframe, the pidapp: tab namespace — is identical either way; a second feature would duplicate all of it for a genuinely small discovery-rule difference (flat file vs. subdir-with-index.html). The cost is real, though: PidApp.root's meaning shifts from "always relative to .pid/" to "relative to .pid/ or the project root depending on source" — a pure-core invariant that today's tests assume implicitly. Budget real test-update time in Phase 3 for this, not just new cases.

2. Creation v1 — scaffold-only, or in-app file editing?

Scaffold-only. In-app editing means the daemon accepts arbitrary HTML/CSS/JS bytes from the browser and writes them to disk — a write-surface increase beyond "create one starter file," and pid-apps' whole security posture to date has been read-only serving of untrusted content. Worth doing later as its own reviewed phase, not folded in here.

3. specs/*.html scan root — hardcoded convention, or configurable via the global-settings file (PR #339)?

Hardcode it, matching .pid/'s own zero-config ethos — one convention, no settings UI to build or document yet. If a second use case ever wants a different root, the global-settings file is where that list would live; not required for this plan.

4. .pid/<name>/ vs. specs/<name>.html id collision — who wins?

.pid/ wins, silently. Same precedent as the existing bare-root-default-vs-subdir rule: locked behavior, not underdetermined. The dropped specs/ entry is simply absent from the list — no error, no warning surfaced to the user in v1.

5. pid-promote — move (rename) the directory, or copy and leave the original in place?

Move. Leaving both would let a stale .pid/<id>/ keep serving the old view-only tab right alongside the new extension, which is confusing and doubles the surface for no benefit. This is a one-way, no-undo filesystem operation run from a CLI outside the dashboard's live-discovery loop — document that plainly in the CLI's own output, not just in this plan.

6. Should specs/-sourced tabs carry a visual "read-only" marker?

Recommend yes if it stays cheap (an icon or small badge) — it prevents a user from expecting a spec doc to be editable the way a hand-authored pid-app might feel. Not worth blocking Phase 3 on if it turns out fiddlier than expected; ship without it and revisit.

7. pid-promote — CLI-only, or a dashboard "Promote" button?

CLI-only for v1, matching pid-ext's own precedent. A dashboard button would still require the user to go restart the daemon by hand afterward — it can't finish the job it starts, so the extra UI surface buys little. Revisit once/if the loader ever supports live reload (deferred in the original pid-apps plan's Questionable #3, still open).

Rejected approach — a separate spec-apps feature

The same reasoning the original pid-apps plan used to reject overloading loader.ts applies here in reverse: pid-apps already owns "discover HTML in a project-scoped, zero-config, manifest-less way." specs/*.html is the same discovery mode, just a second root — not a new mode that deserves its own feature, route family, and tab namespace. A separate feature would duplicate the realpath/CSP/mime pipeline for no real gain.

Engineering guardrails (this repo)

  • TDD always — every behavior change ships a failing test first; the pre-commit tdd-gate enforces it.
  • ts-axioms — Functional Core / Imperative Shell, typed errors, Biome, Lefthook, bun; fallow audit must stay green.
  • Don't split a phase whose shell has no other consumer — the lesson the original plan's Amendments recorded when Phases 4+5 were merged after tripping Fallow's dead-code gate. Applied here twice: Phases 1+2 ship as one PR (createApp has no caller until the route exists), and Phase 3 keeps specs/'s core and repo together in one phase instead of three.
  • CI gates — pre-commit: Biome staged-fix + tdd-gate. pre-push: floor → unit → e2e. Fresh worktree needs bun install first.
  • Restart is a real seam, not a bug — the extension loader is boot-time (platform/extensions/loader.ts, scanRoot at daemon start). pid-promote cannot make a promoted extension appear live; a live-reload loader is out of scope here and was already deferred in the original pid-apps plan's Questionable #3.
  • Ship — one PR per phase; watch CI to terminal state (GitHub auto-merge has been unreliable in this repo).
