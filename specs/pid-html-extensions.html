Purpose

Give every project a frictionless, zero-config way to surface custom HTML inside the dashboard. A user (or an agent) drops an HTML website into the project's .pid/ directory; the dashboard discovers it on demand and renders it as a project-scoped extension tab in a sandboxed iframe. Two concrete use cases drive the design:

  • Specs / plans as HTML — render self-contained HTML documents in-app (this very plan is one such file: see Dogfooding).
  • Run small interactive tools — a static HTML/CSS/JS mini-app (a dashboard, a calculator, a viewer) scoped to one project.

The feature must be secure by default (the dropped HTML is untrusted), per-project isolated (an app in project A never appears for project B), and test-first (TDD red→green→refactor), reusing the proven security and rendering primitives the existing extension platform already ships.

Problem

The repo already has a capable extension platform — but it is the wrong shape for "drop an HTML file and see it." Recon of the existing code shows the loader (apps/daemon/src/platform/extensions/loader.ts) is boot-time, scans a single cwd root (defaultLocalRoot = cwd/.pid/extensions), and requires a manifest.json (scanRoot stats the manifest). That is great for declared, capability-bearing extensions and wrong for an every-project, on-demand, manifest-less drop zone. We need a new thin path — not a fork of the platform, and not an overload of it.

Solution

Add a small, self-contained pid-apps feature that reuses the platform's security + rendering primitives and the existing per-project tab UI, while owning only its own discovery logic:

  • Discovery (zero-config): any immediate subdirectory of <project>/.pid/ that contains an index.html is an app (appId = dirname). A bare <project>/.pid/index.html is the implicit default app (guarded — see Questionables). Reserved names (extensions, extensions-state.json, settings.json) are never apps. Discovery is on-demand per project — no boot scan, no global registry, no manifest.
  • Daemon: GET /projects/:id/pid-apps (list) and GET /projects/:id/pid-apps/:appId/* (serve), mounted on the existing projects router, streaming Bun.file through the same traversal guards as the extensions static route — plus a realpath containment check (symlink escape) and a per-response CSP.
  • Web: a usePidApps(projectId) hook and a thin PidAppHost iframe (sandbox="allow-scripts", opaque origin, no RPC bridge) surfaced as pidapp:<id> tabs in ProjectDashboard, reusing the proven ext:<name> tab machinery.

Questionables open decisions

Surfaced rather than silently decided. Each has a recommended default; confirm before the affected phase.

1. Bare .pid/index.html "default" app — keep the convenience, or require .pid/default/?

Recommend: keep the bare-root convenience, guarded. It is the most literal reading of "drop a HTML website there." But its appRoot is the whole .pid/ dir, which is wider than any other app — so it MUST enforce the RESERVED_PID_ENTRIES exclusion at resolve time (Phase 2). The stricter alternative — require a physical .pid/default/ and drop the bare-root magic — gives a uniform "app = subdir with index.html" invariant with no wide-root special case. If the team prefers the simpler security invariant over the one-file convenience, switch to subdir-only; the core's PidApp.root field already makes the repo branchless either way.

2. Global ~/.pid/ apps as top-level dashboard tabs?

The stated use cases are per-project, so this plan scopes to project tabs only. A homedir ~/.pid/ surfaced as top-level tabs could reuse the same pure core later. Deferred — not required now.

3. Live discovery (fswatch + SSE) vs. React Query polling?

Current plan: low staleTime (~5s) + tab revisit; optionally a one-line manual refresh. An fswatch.repo exists and could emit an ext:state-changed-style SSE to refetch when a file is dropped while the dashboard is open. Recommend deferring live watch to a follow-up; 5s staleTime makes drop-and-see acceptable.

4. CSP 'unsafe-inline' for script-src/style-src — acceptable?

Yes, it is the pragmatic floor. Self-contained HTML (planf3 plans inline everything) cannot carry nonces, and untrusted dropped HTML can't be trusted to. The bound is the opaque-origin sandbox (no allow-same-origin) + connect-src 'none' (no exfiltration). Document the tradeoff at the constant so no one weakens it without understanding it.

5. Asset size cap — 50 MB or tighter?

Recommend 50 MB to match the existing /raw MAX_RAW_BYTES (one constant, no surprise). Revisit only if a tighter HTML-specific ceiling is wanted.

6. Should pid-apps ever opt into the iframe RPC bridge (fs/git capabilities)?

Ship capability-free. These are static sites/specs; no bridge = minimal attack surface. A future opt-in manifest could route through the existing dispatchRpc permission gate with granted empty by default — but only if a concrete interactive .pid tool needs daemon data. Out of scope here.

7. entry restricted to *.html/*.htm only?

Yes. Allowing an SVG (or other script-capable type) as the app entry served as its native MIME bypasses the HTML CSP and can run scripts in some browsers. Constrain entry in the pure parser; non-HTML assets are still served as sub-resources, just never as the entry document.

8. No authentication on the daemon — accepted exposure?

Same posture as the existing /projects/:id/raw and file browser: over a Cloudflare tunnel the routes are reachable by anyone with the URL, and the list endpoint enumerates .pid/ contents. This is a known, pre-existing accepted risk — document it explicitly in AGENTS.md so the exposure is a conscious decision, and ensure no localhost-only assumption creeps into a test.

Rejected approach — overloading the extension platform

We deliberately do not bend loader.ts/registry.ts to do on-demand, per-project, manifest-less discovery. The loader is boot-time, single-root, manifest-required, and carries the permission/RPC machinery — adding a second discovery mode would raise its complexity (Fallow CRAP) and entangle two lifecycles. A thin pid-apps feature that reuses the security/render primitives but owns a tiny pure core is simpler, more testable, and keeps each subsystem cohesive.

Dogfooding

This plan is a single self-contained HTML file. Drop it at <project>/.pid/pid-html-extensions/index.html and, once the feature ships, it renders as a pidapp:pid-html-extensions tab inside the dashboard — the feature displaying its own spec. That round-trip is the acceptance demo.
